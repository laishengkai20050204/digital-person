# 活动现实性后续待办

本文记录活动决策现实性仍需完成的工程项。以下内容尚未实现，不代表当前运行行为。

## 当前已完成

活动决策提示词已经明确要求模型：

- 主动检查 `activeEvents[].elapsedMinutes`；
- 不能因为 `observation` 为空、没有新对话或活动仍存在就默认继续；
- 使用 60～120 分钟作为 `STUDY` / `WORK` 连续专注的重新评估区间；
- 连续超过 180 分钟时，除非上下文有明确截止日期、考试、紧急任务或稳定习惯证据，否则优先结束或中断并休息；
- 结合当地时间、用餐、睡眠、疲劳、困倦、饥饿和日常记忆判断自然活动衔接；
- 让 `nextReviewMinutes` 与活动可能结束的时间匹配。

这些规则属于模型决策约束，不是 Java 硬性时长上限。

## TODO 1：自然生理状态演化

### 问题

当前人物状态主要由事件注册的短期效果推动。若某个活动没有生成完整的生理效果，人物可能长期保持：

- `HUNGER=0`；
- `SLEEPINESS=0`；
- `ENERGY` 不下降；
- `FATIGUE` 不随清醒时间增加。

这会让活动模型接收到不真实的状态，进而合理但错误地维持长时间学习、工作或娱乐。

### 目标

在事件效果之外增加确定性的自然生理演化，使下列变化不依赖模型临时生成：

- 距离上次进食越久，`HUNGER` 逐渐上升；
- 当地夜间和持续清醒时间增加时，`SLEEPINESS` 逐渐上升；
- 长时间清醒或高负荷活动使 `FATIGUE` 上升、`ENERGY` 下降；
- 睡眠使 `SLEEPINESS` 和 `FATIGUE` 下降，并恢复 `ENERGY`；
- 休息使 `FATIGUE`、`MENTAL_LOAD` 等按确定性规则恢复。

### 设计边界

- 优先由 Java 确定性计算，不把基础生理规律交给 LLM；
- 不应重复叠加事件效果已经表达的同一变化；
- 需要明确记录或推导最近进食、最近睡眠、持续清醒时间；
- 所有变化必须支持按时间差一次性结算，无需每分钟写数据库；
- 参数应可测试、可配置，并具有上下限保护。

### 验收标准

- 无活动效果时，长时间未进食仍会增加 `HUNGER`；
- 深夜且持续清醒时，`SLEEPINESS` 会自然增加；
- 完成睡眠后，相关状态会朝恢复方向变化；
- 应用重启后根据持久化时间数据继续正确结算；
- 与现有事件效果同时存在时不会重复计算或越界。

## TODO 2：Java 侧活动持续时长等级

### 问题

当前模型能看到原始 `elapsedMinutes`，但不同模型对“307 分钟学习是否异常”的理解可能不一致。仅依赖提示词和模型常识，稳定性仍然有限。

### 目标

由 Java 根据活动类型和持续时长生成明确的连续性提示，例如：

```json
{
  "activityContinuity": {
    "elapsedMinutes": 307,
    "typicalContinuousMinutes": 90,
    "extendedContinuousMinutes": 180,
    "status": "SEVERELY_EXTENDED"
  }
}
```

建议等级：

```text
NORMAL
EXTENDED
SEVERELY_EXTENDED
STALE
```

### 设计边界

- 这是提供给模型的确定性信号，不直接替模型结束活动；
- 阈值按 `ActivityType` 区分，例如 `EAT`、`STUDY`、`WORK`、`EXERCISE`、`REST`、`SLEEP` 不应共用同一范围；
- 阈值应集中维护并支持配置或领域策略替换；
- `STALE` 可用于极端陈旧事件的恢复和告警，但自动修复策略必须单独设计；
- 截止日期、考试、紧急任务和人物长期习惯仍可作为模型继续活动的反向证据。

### 验收标准

- 同一个活动在不同持续时间下得到稳定等级；
- 不同 `ActivityType` 使用不同阈值；
- 等级进入活动决策上下文并出现在序列化请求中；
- 模型提示词明确要求解释或响应 `SEVERELY_EXTENDED` / `STALE`；
- 不因新增等级破坏事件生命周期和乐观锁语义。

## 建议实施顺序

1. 先实现自然生理状态演化的数据模型和确定性结算；
2. 再增加活动持续时长等级并接入活动上下文；
3. 使用真实调度日志验证学习、吃饭、休息和睡眠的一整天轨迹；
4. 最后再评估是否需要极端时长的 Java 自动恢复兜底。
